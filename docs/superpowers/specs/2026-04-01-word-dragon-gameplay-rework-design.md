# Word Dragon v1 玩法重构设计说明

## 文档信息

- 日期：2026-04-01
- 项目：`word_dragon`
- 目标版本：`v1`
- 范围：关卡页主交互、候选字盘、TTS 提示策略、本地恢复
- 状态：已确认，待用户审阅后进入实施计划阶段

## 背景

当前 `v1` 关卡页虽然已经具备离线加载、候选字输入、提示、通关和本地进度保存能力，但核心玩法交互偏离了常见成语填词游戏的用户心智：

- 页面存在 `当前成语` 和 `成语列表` 两块显式面板，割裂了“在棋盘上解题”的主体验。
- 用户不是通过点击棋盘直接进入输入态，而是被迫理解“先选成语，再从候选字盘输入”的侧向流程。
- 候选字盘没有按剩余可用次数做灰态反馈，不符合填词游戏的常见规则。
- TTS 会直接朗读成语答案文本，破坏猜题过程，使玩法变成“听答案填空”。

本次设计的目标不是增加新玩法，而是把关卡页交互拉回“棋盘驱动”的主流成语填词模式，并确保该模式适合老年用户使用。

## 目标

- 用户只依赖棋盘就能完成整关，不再需要侧向理解“成语列表”。
- 候选字盘要对“剩余可用次数”有明确而即时的反馈。
- TTS 只能提供线索和状态反馈，不能泄露答案。
- 本地恢复不仅恢复盘面，还要恢复当前选中成语和当前输入焦点。
- 保持现有离线、无登录、本地存档、提示、章节解锁等能力不倒退。

## 非目标

- 不修改题库来源、关卡数量和章节体系。
- 不增加联网能力、排行榜、账号和云同步。
- 不引入手写输入、系统输入法输入或语音识别。
- 不增加复杂经济系统，如金币、体力、广告换提示。
- 不在本次设计中重新定义美术风格或主题系统。

## 方案结论

本次采用 `方案 A：纯棋盘驱动`。

结论如下：

- 删除 `当前成语` 和 `成语列表` 两块面板。
- 用户通过点击棋盘上的格子或已存在字块来选择当前要填写的成语。
- 候选字盘基于剩余次数动态置灰。
- TTS 不再朗读成语答案，仅朗读释义提示和状态反馈。

这是最接近市面常见成语填词玩法的方案，也最符合老年用户“看棋盘、点棋盘、填棋盘”的自然操作路径。

## 页面结构重组

关卡页重组后只保留以下结构：

1. 顶部栏
- 返回按钮
- 关卡标题

2. 棋盘区
- 纵横交叉字盘
- 当前选中成语的整条弱高亮
- 当前输入焦点格的强高亮

3. 候选字盘
- 候选字按钮
- 每个候选字的剩余次数状态
- 置灰/禁用反馈

4. 操作区
- `提示一字`
- `揭示成语`
- `发音`
- `自动发音` 开关

5. 通关区
- 仅在通关后出现
- 显示 `进入下一关`

页面上不再显示：

- `当前成语`
- `成语列表`
- 显式暴露成语文本答案的侧栏或卡片

## 主交互模型

### 核心原则

- 棋盘是唯一的解题入口。
- 候选字盘只负责给当前焦点格输入。
- 系统绝不替用户“猜测你想填哪一格”而无提示落字。

### 初始选中规则

进入关卡时：

- 默认选中第一条未完成成语。
- 如果存在本地快照，则恢复：
  - 当前选中成语
  - 当前焦点格
  - 当前盘面字
- 若本地快照里选中的成语已完成，而本关仍有未完成成语，则切到第一条未完成成语。

### 焦点概念

需要引入两个不同层级的选中态：

- `选中成语`
  - 表示当前正在操作哪一条成语
  - 整条成语用弱高亮表示

- `焦点格`
  - 表示下一次候选字输入会落到哪一个格子
  - 仅一个格子可处于焦点态
  - 焦点格高亮强于同条成语其他格子

### 点击普通格

当用户点击仅属于一条成语的普通格时：

- 直接选中该格所属成语。
- 将焦点移动到被点击格。
- 若该格已有字，仍允许作为当前焦点格。

### 点击交叉格

交叉格同时属于横向和纵向两条成语，需要定义明确切换策略。

规则如下：

1. 第一次点击交叉格时：
- 优先选中“未完成”的那条成语。
- 若两条都未完成，则优先选中“更接近可继续输入”的那条：
  - 从点击格开始向后仍有未填或错误格的成语优先。
- 若两条状态完全等价，则优先保持当前方向；若当前没有方向，则优先横向。

2. 再次点击同一个交叉格时：
- 在横向和纵向成语之间切换。

3. 切换方向时：
- 焦点仍停留在该交叉格本身。
- 整条高亮更新为新方向对应的成语。

该规则必须在 UI 和状态机中保持一致，不能让用户点击同一格却无法预测会选中哪一条成语。

## 输入行为

### 输入入口

- 用户只能点击候选字盘输入。
- 输入永远作用于当前 `焦点格`。

### 输入后焦点推进

当用户输入一个候选字后：

1. 当前焦点格写入该字。
2. 如果当前成语中仍有未完成格：
- 焦点自动跳到当前成语中下一个未完成或错误格。

3. 如果当前成语已经填满，但存在错误格：
- 焦点停在该成语第一处错误格。

4. 如果当前成语全部正确：
- 自动切换到下一条未完成成语。
- 焦点落到该成语第一处未完成或错误格。

5. 如果整关全部正确：
- 进入完成态。

### 覆盖输入

- 如果焦点格已有错误字，允许直接覆盖。
- 覆盖时：
  - 被替换的旧字要归还到候选池
  - 新字要从候选池扣减

- 如果焦点格已经是正确字：
  - 再次输入候选字时，不直接覆盖当前正确字。
  - 系统先尝试跳到当前成语中的下一处未完成或错误格。
  - 若没有，则切换到下一条未完成成语。
  - 若整关已完成，则输入无效。

这样可以避免老年用户误触时把已经填对的内容破坏掉。

## 候选字盘设计

### 候选字剩余次数

候选字盘不再只是“本关出现过哪些字”，而是要体现“还剩几个能用”。

计算方式：

`剩余次数 = 关卡原始候选字总次数 - 当前棋盘已占用次数`

例如：

- 某关原始候选池里 `国` 有 `2` 次
- 当前棋盘已经用了 `1` 个 `国`
- 则候选字盘中 `国` 仍可点击，并显示剩余 `1`

### 可点击与灰态规则

- 剩余次数 `> 0`
  - 正常显示
  - 可点击

- 剩余次数 `= 0`
  - 置灰显示
  - 不可点击

### 重复字展示

为了减轻老年用户理解负担，重复字应显示剩余次数角标。

建议表现为：

- `国 ②`
- `一 ③`

若视觉上角标实现成本过高，也至少要保留：

- 置灰逻辑
- 长按语义或无障碍描述里的剩余次数

但推荐首版就把角标做出来，因为这是理解重复字最直接的方式。

### 与棋盘联动

候选字盘必须和棋盘状态实时同步：

- 输入一个字后，候选盘立即刷新
- 覆盖旧字时，旧字立即返还
- `提示一字` 或 `揭示成语` 后，候选盘立即刷新

不允许出现“棋盘已经用了字，但候选盘还亮着”的滞后状态。

## 提示行为

### 提示一字

- 仅作用于当前焦点格
- 若当前焦点格已经正确，则先找到当前成语下一处未完成或错误格，再提示
- 提示后同步刷新候选池剩余次数

### 揭示成语

- 仅作用于当前选中成语
- 将整条成语填写为正确答案
- 同步刷新候选池剩余次数
- 若揭示后本关仍未完成，则自动切到下一条未完成成语

### 提示反馈

提示按钮可以触发反馈语音，但反馈语音不能泄露答案。

允许的反馈示例：

- `已提示一个字`
- `已揭示当前成语`

不允许的反馈示例：

- 直接朗读成语文本
- 朗读完整答案和释义

## TTS 策略重构

### 基本原则

- TTS 是线索系统，不是报答案系统。
- TTS 绝不主动朗读成语答案文本。

### 允许朗读的内容

1. 当前线索提示
- 优先朗读当前成语的释义
- 如果释义为空、过短或不可用，则降级为操作提示，例如：
  - `请填写一个四字成语`

2. 操作反馈
- `已完成一条`
- `已提示一个字`
- `已揭示当前成语`
- `本关完成，可以进入下一关`

3. 异常反馈
- `语音暂不可用`

### 禁止朗读的内容

- 当前成语答案文本
- `成语 + 释义` 的组合朗读
- 通关后逐条播报答案

### 自动发音规则

自动发音开启时，仅在这些时机触发：

- 当前选中成语发生切换时，朗读其释义提示
- 使用提示后，朗读状态反馈
- 通关后，朗读通关反馈

进入关卡时：

- 若恢复的是未完成关卡，可以自动朗读当前已选中成语的释义
- 但不能朗读答案文本

手动点击 `发音` 按钮时：

- 只朗读当前释义提示或当前状态反馈

## 完成态设计

当整关所有成语都填写正确后：

- 棋盘进入完成态
- 候选字盘全部禁用
- 不再允许继续改字
- 显示 `进入下一关`
- TTS 播放：
  - `本关完成，可以进入下一关`

完成态下仍保留棋盘展示，方便用户回看，但不再进入编辑模式。

## 本地存档与恢复

当前存档只恢复盘面还不够，本次需要扩展恢复内容。

### 存档字段扩展

每关快照至少需要包含：

- `levelId`
- `selectedIdiomId`
- `focusedCellCoordinate`
- `cellInputs`
- `hintUsage`
- `isCompleted`

全局进度继续保留：

- `currentLevelId`
- `completedLevelIds`
- `autoSpeakEnabled`

### 恢复规则

重新进入关卡时：

- 若该关未通关：
  - 恢复盘面
  - 恢复选中成语
  - 恢复焦点格

- 若该关已通关：
  - 直接显示完成态
  - 不再恢复到可编辑状态

### 容错规则

若存档中的 `selectedIdiomId` 或 `focusedCellCoordinate` 已失效：

- 优先退化到该关第一条未完成成语
- 若全关已完成，则进入完成态

不允许因为快照失效导致页面无焦点或崩溃。

## 状态模型调整

为了支持新交互，`domain` 和 `ui` 需要补足以下状态。

### Domain 层新增/调整

- 当前焦点格坐标
- 交叉格方向切换规则
- 候选字剩余次数统计
- 正确字保护策略
- 通关后编辑锁定状态

### UI 层新增/调整

- 棋盘格三层状态：
  - 普通格
  - 当前成语弱高亮
  - 当前焦点格强高亮

- 候选字按钮三层状态：
  - 可用
  - 剩余次数提示
  - 不可用灰态

- 去掉基于 `成语列表` 的选择入口

## 错误处理

### TTS 不可用

- 不阻塞游戏
- 显示非阻塞提示
- 用户仍可继续填字

### 无可用候选字

若用户当前焦点格可接受的候选字都已耗尽，但整关尚未完成：

- 系统不自动报错
- 用户仍可通过切换焦点、覆盖错误字或使用提示继续

### 存档损坏

- 忽略非法坐标和非法成语 ID
- 自动回退到首条未完成成语
- 不允许出现页面空白或点击失效

## 验收标准

本次玩法重构完成后，必须同时满足：

1. 页面上不再出现：
- `当前成语`
- `成语列表`

2. 用户只靠点击棋盘就能完成整关。

3. 点击交叉格可以稳定在横/竖两条成语之间切换。

4. 候选字盘会按剩余次数即时灰掉和恢复。

5. TTS 在任何时机都不朗读成语答案文本。

6. 退出重进后，盘面、当前选中成语和焦点格都能恢复。

7. 通关后进入锁定完成态，不能继续编辑。

## 测试策略

### Domain 单元测试

至少新增或更新以下测试：

- 点击普通格后选中正确成语并设置焦点
- 点击交叉格后按规则选择方向
- 重复点击交叉格可切换方向
- 输入候选字后焦点正确推进
- 候选字剩余次数统计正确
- 覆盖错误字时旧字归还、新字扣减
- 正确字保护逻辑生效
- `提示一字` 和 `揭示成语` 会刷新候选池
- 通关后状态锁定
- 快照恢复能恢复焦点格

### UI / 仪器测试

至少新增或更新以下测试：

- 关卡页不再出现 `当前成语` 和 `成语列表`
- 点击棋盘格可以驱动输入
- 灰态候选字不可点击
- 通关后出现 `进入下一关`
- 旋转与重进后焦点和盘面可恢复

### TTS 测试

新增或更新以下断言：

- 自动发音不再包含成语答案
- 手动发音不再包含成语答案
- 只输出释义或状态反馈

## 风险与取舍

### 风险 1：去掉成语列表后，部分用户可能一开始不理解如何切换

应对：

- 首次进入关卡时默认高亮第一条成语
- 焦点格高亮必须足够明显
- 后续如有必要，可在不暴露答案的前提下补一条轻量引导文案，但不在本次首要实现范围内

### 风险 2：交叉格切换规则不稳定会让人困惑

应对：

- 把规则固化在 `domain` 纯逻辑中
- 用单元测试锁住行为
- UI 只消费最终状态，不自行判断方向

### 风险 3：重复字剩余次数表现不清楚

应对：

- 候选字角标优先实现
- 至少保证灰态与点击禁用绝对准确

## 结论

本次玩法重构的核心不是“再加功能”，而是把关卡页交互回归到正确模型：

- 以棋盘为中心
- 以焦点格为唯一输入目标
- 以候选池剩余次数为输入约束
- 以释义提示而非答案朗读作为 TTS 职责

只有在这个模型成立后，`word_dragon` 才真正像一个适合老年用户、且符合常见成语填词直觉的离线游戏。
