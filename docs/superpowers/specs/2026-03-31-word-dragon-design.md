# Word Dragon v1 设计说明

## 文档信息

- 日期：2026-03-31
- 项目：`word_dragon`
- 目标版本：`v1`
- 状态：已确认，可进入实施计划阶段

## 背景

本项目要做一个面向老年用户的 Android 中文成语填词 App。用户群体需要更大的字体、更直接的交互、更少的挫败感，以及离线可用的稳定体验。产品不做登录、不做联网排行榜、不做在线词库更新，首版只聚焦单机闯关体验。

参考实现框架来自 `E:\development\kotlinagentapp`：沿用 Android 原生 Kotlin、Gradle Kotlin DSL、Jetpack Compose 为主的 UI 架构，以及本地单元测试和 Android 仪器测试的组织方式。

## Vision

做一个完全离线、适合老年用户的中文 4 字成语填字游戏，打开即可继续上次进度，也可以进入章节选关。玩家在横屏或竖屏下都能用大号字完成纵横交叉填词，并在点选成语时获得释义提示和清晰的中文语音朗读。首版交付一个可稳定分发的内置常用成语关卡包，规模达到上千关，且每关的排版和触控尺寸都满足可读性要求。

## 产品边界

### 目标

- 完全离线运行，无需登录、注册、联网或账号系统。
- 玩法为纵横填字型成语关卡，每关由 `4` 到 `8` 个四字成语构成。
- 支持横屏与竖屏自动旋转，两套布局都认真适配。
- 默认提供大字号和大触控区域，优先保证老年用户可读性。
- 点选成语时展示简短释义，并自动触发 Android TTS 朗读。
- 提供“揭示一个字”和“揭示整条成语”两类提示。
- 支持本地进度保存、自动续玩、章节选关和已通关关卡重玩。
- 首版内置上千关固定关卡，全部预生成并随 App 打包分发。

### 非目标

- 不做在线对战、排行榜、社交分享、登录账号和云存档。
- 不做运行时动态生成关卡。
- 不做时间限制、体力机制、金币系统、广告驱动的商业化流程。
- 不做 4 字以外的俗语、歇后语、诗词和非成语短语玩法。
- 不做用户自定义词库导入或在线下载关卡。

## 核心玩法

### 关卡结构

- 每关包含 `4` 到 `8` 个四字成语。
- 所有成语必须通过共享汉字形成一个连通的纵横字盘，不允许出现孤立词条。
- 每个成语至少与另一条成语共享 `1` 个字。
- 关卡文件中明确记录每条成语的放置方向、起点坐标、完整答案和提示文本。
- 每关同时记录候选字集合，候选字盘默认只显示本关需要使用的字。

### 玩家流程

1. 从首页点击“继续游戏”或从章节页选择关卡。
2. 进入关卡后看到字盘、当前选中成语、候选字盘和提示按钮。
3. 点击某条成语后，界面高亮对应空格，并显示该成语的简短释义。
4. 同时自动调用 TTS 朗读“成语 + 释义”，并保留重播按钮。
5. 玩家通过底部大号候选字盘逐字填入。
6. 填错时允许直接改，不触发失败状态，也不强制重开。
7. 所有空格填对后自动判定通关，解锁下一关并写入本地存档。

### 输入方式

- `v1` 只使用候选字盘输入，不弹系统输入法。
- 候选字按钮必须满足最小触控尺寸要求。
- 当前选中空格与所属成语需要明确的高亮反馈。
- 删除、清空当前词、撤销上一步为可用操作，但不增加复杂的编辑模式。

### 提示机制

- `提示一个字`：揭示当前选中成语中的一个未填写正确的字。
- `揭示整条成语`：直接填完整条成语。
- 轻松模式下不限制提示次数，但要记录每关提示使用量，供章节页展示“已通关/使用过提示”状态。

### 难度策略

- `v1` 不通过时间或失败惩罚制造难度。
- 难度主要来自成语常用度、字盘密度和交叉复杂度。
- 章节顺序按“常用度低门槛 -> 略复杂 -> 高密度”逐步提升。

## 无障碍与视觉策略

### 排版原则

- 关卡页必须优先保证可读性，不允许为了塞满 `8` 个成语而压缩到难以辨认的字号。
- 横屏是高密度主布局，竖屏是低密度同玩法布局。
- 当某个关卡在某个方向下无法满足最小字号或最小触控尺寸时，该布局必须自动降级为更宽松排版；若仍不满足，关卡生成阶段就视为非法关卡，不打包进 App。

### 最低可用要求

- 字盘汉字显示字号要定义明确下限，并在设计系统中固化。
- 候选字按钮、提示按钮和主要导航按钮要定义明确的最小触控尺寸。
- 颜色和高亮状态不能只靠颜色区分，必须同时有边框、底色或描边变化。
- TTS 朗读失败时要有清晰提示，但不能阻塞游戏。

### 屏幕方向

- App 支持横屏和竖屏自动旋转。
- 横屏布局优先展示完整字盘和候选字盘。
- 竖屏布局保持同样功能，但允许把释义区、候选字盘和辅助按钮重新堆叠排列。

## 内容数据策略

### 成语定义

- 只收录四字成语。
- 不收录网络梗、非固定短语、词语搭配、五字及以上成语、缩略语或诗句。

### 数据来源策略

`v1` 采用“高频筛选 + 可分发词库 + 自有短释义”的组合策略：

- 使用 `pwxcoo/chinese-xinhua` 中可分发的成语数据作为基础词库来源，用于获得四字成语、拼音和基础释义素材。
- 使用 `THUOCL` 的高频语料词表为成语做常用度排序和筛选。
- 不直接打包教育部《成语典》原始释义文本，因为其授权和改编限制不适合直接作为 App 内文案分发方案。
- App 最终内置的是经过筛选、清洗和统一风格改写后的成语数据包，其中提示文本使用短句风格，长度受 UI 限制控制。

### 成语词条字段

每条可玩成语至少包含：

- `id`
- `text`
- `pinyin`
- `short_explanation`
- `tts_text`
- `frequency_rank`
- `difficulty_tier`
- `enabled`

### 关卡数据字段

每个关卡至少包含：

- `level_id`
- `chapter_id`
- `idiom_ids`
- `board_width`
- `board_height`
- `placements`
- `candidate_chars`
- `orientation_hints`
- `layout_profile`

### 存档字段

本地存档至少包含：

- `current_level_id`
- `unlocked_level_id_max`
- `level_completion_states`
- `level_hint_usage`
- `in_progress_board_snapshot`
- `last_played_at`

## 关卡生成策略

### 总体方案

`v1` 使用预制关卡方案。所有关卡在开发期由离线脚本批量生成、校验和筛选，通过后打包进 App。运行时不做动态出题。

### 生成约束

- 每关成语数量必须在 `4-8` 之间。
- 字盘必须连通。
- 不允许重复成语。
- 不允许存在未被其他成语交叉的孤立词条。
- 同一章节内不允许出现高重复字符结构导致的视觉混淆关。
- 每关都要通过布局校验：字号下限、触控下限、横屏布局、竖屏布局。

### 质量门槛

- 关卡生成脚本输出的候选字数量不能过多，避免字盘之外的干扰字符泛滥。
- 同一关卡中若存在多个几乎同义、极易混淆的生僻成语组合，则直接淘汰。
- 默认优先选择高频、解释清楚、形近字干扰可控的成语。
- 对极少数自动生成质量不稳的关卡，允许人工审核并剔除。

## 技术架构

### 平台与依赖

- Android 原生 Kotlin
- Gradle Kotlin DSL
- `minSdk 24`
- `targetSdk 35`
- Jetpack Compose 为主
- 使用 Android TTS
- 单元测试采用 JUnit + Robolectric
- 界面流程验证采用 `androidTest`

### 分层

#### `data`

- 负责加载内置成语 JSON、关卡 JSON、章节索引和本地存档。
- 负责 JSON 与领域模型映射。
- 不承担玩法判定逻辑。

#### `domain`

- 负责关卡状态机、填字判定、提示消耗、通关检测、章节解锁和续玩逻辑。
- 负责把成语词条转成 UI 所需的提示和 TTS 文案。
- 设计为纯 Kotlin 逻辑，优先单元可测。

#### `ui`

- 负责首页、章节页、关卡页、设置页和通关反馈。
- 只消费 `domain` 层的状态与事件，不直接读写原始 JSON。

#### `tts`

- 负责 TTS 初始化、朗读、停止和异常处理。
- 复用参考工程里已经验证过的 Android TTS runtime 设计模式，包括 provider 封装、协程化朗读接口和失败降级处理。

#### `tooling`

- 存放词库清洗、常用度筛选、关卡生成和布局校验脚本。
- 这些脚本是构建高质量固定关卡包的关键资产，不放进运行时路径。

## UI 结构

### 首页

- 主按钮：`继续游戏`
- 次按钮：`章节选关`
- 附加入口：`设置`
- 首页直接展示最近进度摘要，例如当前章节、当前关和是否有未完成盘面。

### 章节选关页

- 按章节展示已解锁与未解锁关卡。
- 已通关关卡需要显示通关状态和是否使用过提示。
- 点击已通关关卡允许重玩。

### 关卡页

- 字盘区域
- 当前成语释义区域
- 候选字盘
- 提示区和朗读重播按钮
- 返回与重置当前关卡入口

### 设置页

- 朗读开关
- 朗读语速基础档位
- 字号档位选择
- 是否默认自动朗读当前成语

`v1` 不做复杂个性化设置，不做主题皮肤商城，不做账号相关设置。

## TTS 设计

- 默认行为：用户点选某条成语时自动朗读。
- 朗读内容：优先为“成语 + 简短释义”。
- 提供手动重播按钮。
- 当系统 TTS 不可用、未安装中文语音或初始化失败时，游戏仍可继续，只展示非阻塞错误提示。
- TTS 不能和核心填字逻辑强耦合，必须可关闭和可替换。

## 本地存档设计

- 存档默认自动写入，不要求用户手动保存。
- 进入关卡、填字、使用提示、通关和切后台时，都要触发安全写盘或合并写盘策略。
- 重新打开 App 后，首页“继续游戏”应恢复到上次未完成关卡及其盘面状态。
- 已通关记录和解锁进度必须稳定恢复。

- 成语词条过滤规则
- 关卡状态机
- 填字正确/错误判定
- 提示一个字与揭示整条成语

### 关卡资源校验

- 章节索引和关卡文件引用完整
- 成语数据和关卡数据之间没有断链

## 风险与应对

### 风险 1：词库常用度不足，导致老年用户体感偏难

应对：

- 以高频排序为前置过滤条件。
- 为前期章节单独设定更严格的常用度阈值。
- 对短释义风格进行统一改写，减少理解成本。

### 风险 2：单关 `8` 个成语与大字要求冲突

应对：

- 把最小字号和最小触控尺寸前置为关卡生成约束。
- 横屏和竖屏分别校验，不合格直接淘汰。
- 不以“凑满 8 个”为目标，允许部分关卡保持 `4-6` 个成语。

### 风险 3：Android TTS 在不同设备上表现差异较大

应对：

- TTS 设计成可关闭的增强功能。
- 初始化失败不阻断主流程。
- 文案长度控制在短句范围，避免长文本朗读体验不稳定。

### 风险 4：上千关固定内容的质量控制成本高

应对：

- 优先建设关卡校验脚本。
- 把“非法布局淘汰”和“低质量词条淘汰”自动化。
- 保留小规模人工抽检环节，但不依赖人工逐关制作。

## 方案结论

`word_dragon` 的 `v1` 采用“预制关卡优先，词库策展，完全离线，轻松闯关”的路线。产品体验优先级高于题量炫技和商业化机制，任何实现都不能牺牲大字、清晰、高可读性和稳定续玩。后续实施计划将围绕 `M1-M4` 展开，并把词库合法分发、固定关卡质量和老年用户可读性作为核心验收轴。
