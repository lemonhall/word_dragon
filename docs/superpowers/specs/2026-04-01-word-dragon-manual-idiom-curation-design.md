# Word Dragon M5 人工成语审核流水线设计

## 背景

`Word Dragon v1` 当前已经具备离线运行、关卡生成、棋盘交互、TTS、进度恢复和发布前验证能力，但词库层仍存在一个根本问题：现有 `36084` 条词条只是按“四字 + 常用字 + 有释义”筛出来的，并不等于“适合老年用户游玩的常用成语”。

这一点已经在现有内容包中暴露出来。虽然 `1000` 关只实际使用了 `1451` 条唯一成语，但其中仍出现了明显不适合作为大众成语游戏核心词库的条目，例如 `一世两清`、`一世之利`、`一世龙门`、`一了百当` 等。问题不在字频，而在“成语本身是否常用、是否顺手、是否适合老年用户理解和联想”。

因此，`M5` 不再尝试用算法继续逼近“常用成语”，而是建立一条严格受控的人审流水线：冻结当前 `36084` 条词条，按固定批次逐条人工审核，最终产出一份真正可追溯的常用成语辞典。

## 目标

- 以当前 `app/src/main/assets/content/idiom_catalog.json` 中 `enabled=true` 的 `36084` 条词条为唯一审核源。
- 将全部词条按固定顺序切成 `181` 批，前 `180` 批每批 `200` 条，最后一批 `84` 条。
- 建立一套人工审核工作区，使每一条成语都必须逐条给出 `keep` 或 `filter` 结论。
- 建立 harness，确保整个审核过程不可跳批、不可漏审、不可重复、不可伪造源条目。
- 生成最终“人工审核后的常用成语辞典”，供后续内容管线和关卡重建使用。

## 非目标

- `M5` 不直接修改 Android 端玩法、界面或 TTS 行为。
- `M5` 不在审核中途回灌词库并重新生成关卡。
- `M5` 不引入基于算法或脚本的“自动保留 / 自动过滤”判断。
- `M5` 不追求保留尽可能多的词条；用户体验优先于数量。
- `M5` 不把“人工审核”扩展成百科式编纂，不补写长释义或词典考据。

## 现状事实

- 当前源词库路径：`app/src/main/assets/content/idiom_catalog.json`
- 当前启用词条数：`36084`
- 当前关卡总数：`1000`
- 当前已被关卡使用的唯一成语数：`1451`
- 当前 PRD 中的 `REQ-0001-002` 要求“仅使用常用四字成语”，但现有实现只能证明“仅使用常用字构成的四字词条”，尚不能证明“成语本身常用”

## 方案对比

### 方案 A：固定批次 + JSONL 审核记录 + Harness 状态机（推荐）

- 冻结源词表后按顺序切成固定批次。
- 每批生成只读 `source` 文件，人工逐条填写对应 `review` 文件。
- Harness 校验通过后自动产出该批 `approved` 文件，并推进总进度。
- 优点：强约束、可恢复、可追溯、便于长周期执行。
- 缺点：前期要先搭完整的审核工作区和状态约束。

### 方案 B：纯 Markdown 人工表格审核

- 直接生成 `181` 份 Markdown 表格，人工在表格中填写结论。
- 优点：直观，可直接阅读。
- 缺点：极难强约束；防漏、防重、防跳批和最终汇总都不可靠。

### 方案 C：单大文件顺序审核

- 不拆批，直接对 `36084` 条总表从头审核到尾。
- 优点：结构最简单。
- 缺点：恢复点太粗，人工执行压力大，长周期中断后极易失控。

## 推荐方案

采用方案 A。

这次任务的核心不是“把文件拆开”，而是“建立一个能支撑 3.6 万条人工审核且不失控的流水线”。固定批次、固定顺序、只读源文件、人工审结文件和强校验 harness 是必要条件，不是附加项。

## 设计

### 1. 冻结审核源

审核源固定为当前 `idiom_catalog.json` 中全部 `enabled=true` 的词条，按现有 `entries` 顺序冻结，不再按字频、释义长度或任何额外规则重排。

冻结后，每条记录必须获得一个稳定的全局序号：

- `global_seq`：`1..36084`
- `idiom_id`：沿用已有 `idiom-xxxxx`
- `text`
- `pinyin`
- `short_explanation`

这一步的作用是把“本轮到底审的是哪一版词库”彻底钉死。后续任何审核记录都必须逐条对齐这份冻结源。

### 2. 工作区结构

`M5` 审核工作区放在：

`research/content/manual_review/m5/`

目录结构如下：

```text
research/content/manual_review/m5/
├── source/
│   ├── master_catalog.jsonl
│   ├── batches/
│   │   ├── batch-0001.source.jsonl
│   │   ├── batch-0002.source.jsonl
│   │   └── ...
│   └── manifest.json
├── reviews/
│   ├── batch-0001.review.jsonl
│   ├── batch-0002.review.jsonl
│   └── ...
├── approved/
│   ├── batch-0001.approved.jsonl
│   ├── batch-0002.approved.jsonl
│   └── final_common_idioms.json
├── audit/
│   ├── progress.json
│   ├── batch_reports/
│   │   ├── batch-0001.report.json
│   │   └── ...
│   └── source_hashes.json
└── docs/
    ├── m5-review-guideline.md
    └── m5-execution-notes.md
```

其中：

- `source/` 只生成一次，后续不允许手改。
- `reviews/` 是人工逐条审核输入。
- `approved/` 由 harness 根据已过检的 `reviews/` 自动生成，不允许人工直接编辑。
- `audit/` 记录进度、批次状态、校验结果和源文件哈希。
- `docs/` 保存审核标准与执行说明。

### 3. 批次规则

固定拆成 `181` 批：

- `batch-0001` 到 `batch-0180`：各 `200` 条
- `batch-0181`：`84` 条

每批固定保留两个序号：

- `global_seq`：源总表中的绝对位置
- `batch_seq`：该批内的 `1..N`

拆批后，任何后续处理都不能改变批次边界、顺序或条目内容。这样可以确保整个审核过程是“从 `1` 号词条走到 `36084` 号词条”，而不是一轮轮“动态挑样”。

### 4. 文件格式

#### 4.1 源文件

`batch-0001.source.jsonl` 每行一条，字段如下：

```json
{"global_seq":1,"batch_seq":1,"idiom_id":"idiom-00001","text":"一世两清","pinyin":"yī shì liǎng qīng","short_explanation":"旧指兄弟两人都是清明廉正的官吏。"}
```

#### 4.2 审核文件

`batch-0001.review.jsonl` 也必须逐行一一对应：

```json
{"global_seq":1,"idiom_id":"idiom-00001","text":"一世两清","decision":"filter","note":"偏冷僻，现代大众语感极弱，不适合作为常用成语游戏词条。"}
```

约束如下：

- `decision` 只允许 `keep` 或 `filter`
- `filter` 时 `note` 必填
- `keep` 时 `note` 可为空
- `global_seq`、`idiom_id`、`text` 必须与源文件完全一致

#### 4.3 过审文件

`batch-0001.approved.jsonl` 不由人工填写，而是由 harness 从 `review` 中抽取 `decision=keep` 的记录生成。这样可以从机制上防止“跳过 review 直接写 approved”。

最终总表为：

- `approved/final_common_idioms.json`

该文件由所有 `batch-xxxx.approved.jsonl` 汇总生成，并保留源字段和审核元数据。

### 5. Harness 设计

Harness 只负责编排、校验、汇总和报表，明确禁止参与“常用 / 生僻”的判断。

推荐命令边界如下：

- `prepare`
  - 从当前词库生成冻结总表、批次源文件、总 manifest、源文件哈希清单
- `status`
  - 输出当前总进度、当前可开批次、已完成批次、待审条数
- `open-next`
  - 只允许打开下一个未完成批次；前一批未闭环时拒绝继续
- `validate-batch --batch batch-0001`
  - 检查该批 `review` 是否完整、合法、逐条对齐源文件
- `close-batch --batch batch-0001`
  - 仅在 `validate-batch` 通过后生成 `approved` 文件和批次报告，并更新总进度
- `assemble`
  - 仅在全部 `181` 批完成后汇总最终词典

### 6. Harness 的硬约束

`validate-batch` 必须同时验证以下事实：

- 本批 `review` 条目数与源批次数完全一致
- 每条 `global_seq`、`idiom_id`、`text` 都与源批次逐条对应
- 不允许缺条
- 不允许多条
- 不允许重复序号
- 不允许无效 `decision`
- 不允许 `filter` 无原因
- 不允许源文件哈希变化

`close-batch` 必须拒绝以下情况：

- 前置批次尚未完成
- 当前批次未通过校验
- 当前批次已经关闭却又试图重复关闭

这就是本次任务的反作弊核心：Python 可以负责“审计”，但不能替代“裁决”。

### 7. 人工审核规则

人工审核判断围绕一个问题展开：

“这条四字成语是否适合进入面向老年用户、强调常用与顺手的大众成语填词游戏？”

以下类型倾向直接 `filter`：

- 明显冷僻，现代日常几乎不用
- 更像书面典故、古籍用语、僻义官样语、宗教术语或历史专名
- 释义即使简化后仍然难懂，普通用户难以从提示反推出答案
- 虽然形式上是四字，但不属于大众语感中常见的“成语游戏词条”

以下类型倾向 `keep`：

- 普通用户在成语游戏、日常表达或媒体语境里能较自然识别
- 释义容易理解，和答案之间的联想路径明确
- 适合作为老人向文字游戏内容，不会让玩家频繁产生“这也算成语？”或“这也太生僻了”的挫败感

### 8. 审核状态边界

本轮不引入 `待复核` 第三状态。

原因有两点：

- `36084` 条已经足够长，第三状态会把主流程拖成“初审 + 积压池”，导致长期失控
- 用户当前优先级是“先建立一条严格可执行的全量人工审核流程”，而不是先做复杂的复审分叉

边界词条的处理方式是：仍然给出 `keep` 或 `filter`，并把判断理由写进 `note`。如果后续确实需要二轮复审，再基于首轮 `note` 做专题返工。

### 9. 进度与留痕

`audit/progress.json` 必须至少记录：

- 总词条数
- 总批次数
- 当前已完成批次数
- 当前已审核条数
- 当前保留条数
- 当前过滤条数
- 当前可开批次编号
- 每批状态：`pending / in_review / validated / closed`

每个批次完成后生成：

- `audit/batch_reports/batch-0001.report.json`

其中记录：

- 该批条目数
- `keep` 数量
- `filter` 数量
- 审核开始时间
- 审核关闭时间
- 校验通过时间
- 源文件哈希
- 审核文件哈希

这样后续任何人都能回答三个问题：

- 现在审到哪一批了
- 某一批到底有没有完整审完
- 最终某条成语为什么被保留或过滤

### 10. 最终回灌边界

`M5` 的直接交付物是“人工审核后的常用成语辞典”，不是新的游戏关卡包。

后续应单开下一轮内容里程碑，做以下事情：

- 用 `final_common_idioms.json` 替换现有自动筛选词库作为生成源
- 重新设计关卡生成策略，提高唯一成语覆盖率
- 重新生成 `1000+` 关
- 重新运行内容校验、布局校验和真机验证

把“人工审词”与“内容重建”拆开，可以避免在审核尚未闭环时反复扰动游戏内容。

## 测试与验证策略

`M5` 虽然不做玩法实现，但仍需自动化验证其流程纪律。

至少需要覆盖以下层面：

- `prepare` 能稳定生成 `181` 批，且总量为 `36084`
- `validate-batch` 能拦截缺条、多条、重复、无效 `decision`、`filter` 无原因、源内容不匹配
- `close-batch` 只能在前序批次闭环后执行
- `assemble` 只有在全部批次完成后才允许运行
- 任意抽查一条最终保留词条，都能回溯到源批次与审核记录

## 风险与约束

- 最大风险不是实现，而是执行周期极长；如果没有强进度台账和硬状态机，人工审核中途必然失控。
- 如果允许人工直接修改 `approved` 文件，整个追溯链会失效，因此必须把 `approved` 设为派生产物。
- 如果后续更换源词库但不重建 `manifest` 与哈希，之前所有审核结果都会失去意义，因此冻结源是强前提。
- 如果后续发现二分类不足，再单独开复审机制；本轮不能先把流程做复杂。

## 结论

`M5` 的本质不是“继续调一个筛词脚本”，而是把“常用成语”从算法近似问题，转换为一条严格受控、全量留痕、可长期执行的人审生产线。

先把这条生产线搭稳，再进入真正的全量人工审词，后续内容质量才有机会从根上稳定下来。
