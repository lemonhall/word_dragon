# Word Dragon 自动切换到下一成语时补播释义设计

## 背景

`ECN-0003` 已经补上“成语完成时播答案”，但继续试玩后又出现一个新的断点：当系统因为当前成语完成而自动把焦点切到下一条未完成成语时，界面上下文已经变了，语音却没有同步补一遍新成语释义；用户只有再次点击棋盘，才会听到它。

这会把“开始新的成语填词工作”拆成两步：视觉上已经切换，听觉上却还停在上一条成语的完成反馈。对依赖语音提示的老年用户来说，这个节拍是断的。

## 目标

- 当系统自动切换到下一条未完成成语时，自动补播一次该成语释义。
- 现有“成语首次完成时播答案”保留，且播报顺序清晰稳定。
- `重播发音` 的语义不变，仍然只重播当前释义。

## 非目标

- 不改动 TTS 文案格式。
- 不新增新的语音设置项或播报队列配置。
- 不改动 `GameSessionEngine` 的棋盘选择与焦点推进规则。

## 方案对比

### 方案 A：在 `GameViewModel` 基于 `selectedIdiomId` 变化补播下一条释义（推荐）

- 先沿用现有“新完成成语集合”逻辑播答案。
- 再比较前后 `selectedIdiomId`，如果本次操作让系统自动切到另一条未完成成语，就补播这条新成语释义。
- 优点：改动面最小，不动状态机和 TTS 格式器；手工填字、`提示一字`、`揭示成语` 三条路径都能复用同一触发逻辑。
- 缺点：`GameViewModel` 需要同时判断“是否切换了成语”和“是否还有未完成目标”。

### 方案 B：在 `GameSessionEngine` 显式返回一个“下一个要播的语音事件”

- 由领域层把“完成播报”和“自动切换播释义”都编码为事件，再由 `GameViewModel` 消费。
- 优点：语音事件边界更显式。
- 缺点：会扩大状态机和返回模型的改动范围；对这次单点体验修补来说过重。

## 推荐方案

采用方案 A。当前问题不在领域状态错误，而在 UI 层没有把“自动切换到下一条成语”当成一次应播释义的上下文切换。把修复放在 `GameViewModel`，可以最小代价补齐行为，同时保留现有测试和状态机稳定性。

## 设计

### 1. 播报顺序

- 当一次操作让某条成语首次完成时，先播该成语答案。
- 如果同一次操作又把选中成语切到下一条未完成成语，再紧接着播这条新成语释义。
- 如果本次操作直接通关，没有“下一条未完成成语”，则不追加新的释义播报。

### 2. 触发条件

- 比较 `currentSession.selectedIdiomId` 与 `nextSession.selectedIdiomId`。
- 只有在以下条件同时满足时，才补播下一条释义：
  - 自动发音开关开启；
  - 前后 `selectedIdiomId` 不同；
  - `nextSession` 未通关；
  - 新选中的成语在 `nextSession` 中仍未完成。

### 3. 接口与数据

- 不新增新的持久化字段。
- 不改 `GameSpeechFormatter`；下一条成语释义仍走现有 `format()`。
- 不改 `GameSessionEngine` 返回结构；仍由 `GameViewModel` 从前后状态差异推导播报行为。

## 测试策略

- 新增 `GameViewModelSpeechFlowTest` 仪器测试，直接验证 `GameViewModel` 的播报顺序：
  - 手工填完当前成语并自动切到下一条时，播报顺序为“初始释义 → 完成答案 → 下一条释义”。
  - `揭示成语` 让当前成语完成并自动切到下一条时，也满足同样顺序。
- 现有 `PlayLevelFlowTest` 保留，继续验证单成语关卡和页面文案，不把这个多成语播报边界强塞进原测试。

## 风险与约束

- 若只判断 `selectedIdiomId` 变化、不判断 `nextSession.isCompleted`，则最后通关时可能错误补播无意义的释义。
- 若把“下一条释义”放在“完成答案”之前，会削弱当前成语的完成反馈，不符合上一轮已确认的体验。
